1. display:none에 애니메이션 적용불가문제

보고서를 보시면 아시겠지만, 제 웹사이트 오른쪽 하단에는 토글 메뉴가 존재하고 있습니다. 문제는 여기서 발생했는데요,
저는 이 홈페이지의 어디서든 이 메뉴를 클릭하면 원하는 메뉴와 소제목으로 이동할 수 있도록 만들고 싶었습니다.
또한 해당 메뉴란이 나타날때, transition으로 애니메이션을 주어 부드럽게 나타날 수 있도록 하고 싶었죠.
하지만 왜인지는 모르게 적용이 되지는 않았습니다.

이를 해결하기 위해서 구글에 검색도 해보고, 우리의 chatgpt와 같은 인공지능에게도 물어보았습니다. 답은 의외로 간단했습니다.
바로 이 display: none;이 display: block;으로 스타일 변환될때는, 애니매이션이 적용되지 않는다는 것이었습니다.

display속성은 콘텐츠를 시각적으로 사라지게 만들지만, 화면에 요소를 랜더링하는 랜더링 트리도 같이 숨겨지는 특징이 있습니다.
그래서 transition을 시도할 때 요소의 처음 상태를 알 수가 없었기 때문에 상태 변화간 애니메이션을 줄 수가 없었던 것입니다.

따라서 저는 이 문제를 해결하기 위해서 display속성 대신 visibility속성을 사용하였습니다.



visibility속성은 display처럼 요소를 시각적으로 숨기는 속성이지만, 요소의 랜더링 트리 자체를 숨기지는 않습니다.
그말인즉, transition속성이 정상적으로 작동한다는 이야기라고 할 수 있겠지요.

이 속성을 사용하여 저는 제가 원하는 효과를 구현할 수 있었으며, 하나의 지식을 더 알아간다는 느낌을 받을 수 있었습니다.

2. canvas 화질 저하 문제

제 메뉴바의 마지막을 보시면 Future Plan이라는 탭이 있습니다.
이 탭에서는 저의 미래 계획을 선과 이미지, 도형을 통해서 보여주고 있는데, canvas를 통해서 작성되었습니다.
이 부분을 코딩할 때 문제가 발생하였는데, 그것이 바로 canvas화질 저하 문제였습니다.

future plan탭을 보시면, 상당히 넓은 범위의 canvas가 보이실겁니다. 이 canvas는 html환경에서 그래픽 기능을 구현할 수 있도록 도와주는 태그입니다.
canvas 그리기 영역의 기본값은 300x150으로 구성되어 있는데, 문제는 바로 여기서 발생하였습니다.
캔버스 요소의 크기를 늘린다고 이 '그리기 영역'의 크기가 같이 늘어나지는 않는다는 것. 그것이 제가 겪은 문제입니다.



그림에서 보시다시피 css에서 캔버스 요소만 크기 설정을 해주고, 그것이 그리기 영역의 기본값과 일치하지 않는다면 브라우저의 캔버스 표시 방법은 다음과 같습니다.
1. 그리기 영역의 기본값인 300x150에 맞추어 그림을 그린다.
2. 브라우저가 그리기 영역의 크기를 그래도 변형시켜 캔버스 영역에 맞춘다.
이렇게되면 그림의 픽셀들의 크기가 증가하게 되어 화질이 저하되는 문제가 발생했던 것입니다.

해결방법은 무엇이었을까요? 살펴보니 답은 간단했습니다.
css를 통해 캔버스 요소의 크기를 변형시키지 않고, html에서 직접 폭과 높이를 성정하면 그리기 영역의 크기도 같이 늘어난다는 것이었습니다.



따라서 해당 캔버스의 속성을 직접 변경함으로써, 화질이 깨지지 않는 도형과 선을 그릴 수 있게 되었습니다.

3. 로컬 스토리지의 파일 표시 에러 문제

저의 relation companies탭을 보시면, 자신이 관심이 가는 회사의 정보를 추가할 수 있는 입력창을 불러올 수 있습니다.
해당 입력창의 로고 이미지 입력 부분은 초기에는 input태그의 file타입을 이용하였으나, 곧이어 문제를 발견하게 되었습니다.
바로 해당 기능으로 읽어온 로고 이미지는 제대로 html에 표시되지 않는다는 것이었습니다.

처음에는 이게 왜 작동하지 않는거지?라고 궁금함을 참지 못했습니다.
저장소에 있는 파일을 불러와 웹사이트에 적용시킬 수 있다면 제일 베스트라고 생각했기 때문입니다.
하지만 조금 찾아보니, 그것은 현재 이 웹사이트의 수준으로는 불가능하다는 것을 깨달았습니다.

input의 file 태그는 사용자의 저장소에 있는 파일을 읽어오는만큼, 보안에 관련된 문제가 발생할 가능성이 큽니다.
따라서 해당 태그는 파일을 읽어올때 외부에서 그 경로를 알 수 없도록 fakepath를 사용하는데, 이것 때문에 웹사이트가 파일을 읽어도 이미지가 표시되지 않았던 것입니다.
따라서 저는 이것을 해결하기 위해서, 로컬 사용자의 저장소가 아닌 웹에 저장되어 있는 이미지의 주소를 받아오는 형태로 코드를 수정하였습니다.



이렇게 웹 url을 받아서 이미지를 표시하면 보안에 위협이 될 걱정도 없고, 정확성이 떨어지는 사용자 파일보다는 해당 회사 홈페이지에 있는 공식 이미지를 사용하는 방법이 더 괜찮다는 생각이 들어서 바꾸게 되었습니다.
앞으로 이 부분은 웹 서버를 구축하여 이미지를 저장하거나, database를 구축하여 다른 사용자들도 해당 이미지를 확인할 수 있는 방향으로 발전시켜나갈 계획입니다.
